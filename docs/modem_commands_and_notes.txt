Windows run w/ debug and force protocol from command line example:
$env:V32_DEBUG="1";node .\src\index.js --force-proto V22 2> server-debug.log

What can you get out of it diagnostically? Many modern modems support:

ATW2 or similar verbose connect messages showing which protocol actually negotiated
AT#UD or vendor-specific "last call stats" showing why training failed
Speaker enable (ATM2) so you can hear what the modem hears.
ATI6 / ATI7 / ATI10 / ATI11 — these are all vendor-specific "extended info" commands.
AT-K1 or AT-SR=1 — enables "trace mode" on some Rockwell/Conexant modems
AT&V — dumps current S-register values and config
ATW3 (if supported) — verbose connect message showing training phases
Speaker on: ATM2L3 — highest speaker volume, stays on throughout call.
