# vm/overlay/ — Static files injected into the runtime initramfs

This directory tree is copied verbatim into the staged rootfs during
`make populate` (see `vm/Makefile`). Its layout is just a relative
path under the rootfs root: `overlay/etc/passwd` becomes `/etc/passwd`
inside the VM.

Everything here is version-controlled, human-readable, and directly
editable. If you need to change behavior at the guest-userspace
boundary, edit files here and rebuild — `make -C vm` picks up
changes automatically.

## Files

### `etc/passwd` and `etc/group` — the drop-privs mapping

**Both `nobody` and `nogroup` are mapped to uid/gid 0.** This is
deliberate.

slmodemd upstream always drops privileges to user `nobody` when
it's started as root. The relevant code in `modem_main.c`:

```c
#define SLMODEMD_USER "nobody"
...
if (getuid() == 0) {
    pwd = getpwnam(SLMODEMD_USER);
    ...
    setgroups(1, &pwd->pw_gid) || setgid(pwd->pw_gid) || setuid(pwd->pw_uid);
    ...
}
```

On a general-purpose Linux system this is a sensible precaution —
soft-modem code touching audio buffers at high priority shouldn't
have root. Our VM is different:

- No network services. No sshd, no getty, nothing listening.
- No other users. root is the only login.
- Read-only initramfs. The filesystem resets on every boot.
- Two virtio-serial channels to the host are the only external
  surface, wired directly to our trusted Node process.

Dropping privileges in this environment achieves nothing and
actively creates problems:

- slmodemd creates `/dev/ttySL0` as `nobody` (mode 0660), so the
  shim — which forked from slmodemd and inherited the drop —
  opens it fine, but *future* processes spawned as root can't.
- Device nodes under `/dev/virtio-ports/` default to mode 0600
  root:root, so the dropped-privs shim gets EACCES trying to
  open them.
- Each of these needs a separate workaround (chmods, group
  membership, etc.) for no benefit.

Rather than fork slmodemd to remove the drop-privs code — which
we explicitly refuse to do (see `IMPLEMENTATION.md`) — we make
the drop a no-op by mapping `nobody` to uid 0 in our `/etc/passwd`.
When slmodemd calls `setuid(pwd->pw_uid)` where `pw_uid = 0` while
already root, it's a no-op.

This is a deliberate security non-posture that relies on the
appliance-VM threat model. It would be completely inappropriate
on a multi-user system. In our VM it's the cleanest solution.

`dialout` (gid 20) is kept at its conventional value because
slmodemd looks it up by name (not gid), and the PTY link is
chgrp'd to dialout when it's created. Having the group exist
satisfies the lookup; the gid value itself is unused here.

### `etc/init.d/rcS`

Busybox init's `::sysinit:` entry — runs before any respawn
entries fire. Mounts `/proc`, `/sys`, `/dev`, `/dev/pts`, `/tmp`
so S99modem can assume they're up. Then runs any other `S??*`
scripts in the init.d directory (currently none beyond S99modem,
which is launched separately by inittab on ttyS0).

### `etc/init.d/S99modem`

The service script for slmodemd. Loads virtio kernel modules,
enumerates `/sys/class/virtio-ports/vport*/name` to create
`/dev/virtio-ports/<name>` symlinks (our udev replacement), and
execs slmodemd with the shim as its -e child.

Respawn mode: if slmodemd exits, busybox init restarts S99modem
automatically via `/etc/inittab`'s `ttyS0::respawn:` entry.

## Adding new files

Anything that needs to exist in the runtime rootfs should live
here. Examples of what *could* go here but currently doesn't:

- `/etc/resolv.conf` — none needed, no networking.
- `/etc/hosts` — none needed.
- `/etc/localtime` — the VM uses UTC; no timezone required.

Don't add files here that are generated by the build — those
belong in `vm/Makefile`'s populate target (examples: `/etc/inittab`,
`/etc/fstab`, `/etc/hostname`).
